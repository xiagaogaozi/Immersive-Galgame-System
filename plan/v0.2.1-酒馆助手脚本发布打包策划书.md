## 数据流/接口契约

- 输入来源：
  - IGS 主程序源码：`app/src/**`
  - 构建脚本：`app/scripts/build.js`
  - 版本信息：`app/package.json` 与 `app/dist/manifest.json`
  - 原版参考：`../Visual Novel/` 与 `_inbox/酒馆助手脚本-玉子手机.json`
- 处理过程：
  - `npm run gate` 先跑完整模拟闸门。
  - `npm run build` 生成 `app/dist/*`。
  - 后续发布阶段新增 `loader/igs-loader.js`，再把它嵌入 `loader/igs-loader.json` 的 `content` 字段。
  - loader JSON 的结构遵循 JS-Slash-Runner：`type`、`enabled`、`name`、`id`、`content`、`info`、`button`、`data`、`export_with`。
- 输出目标：
  - 本地可导入文件：`loader/igs-loader.json`
  - 远程 bundle：`app/dist/igs.bundle.js`、`app/dist/igs.bundle.css`
  - GitHub 上传目标：`https://github.com/xiagaogaozi/immersive-galgame-system`

## 通过标准

- README、AGENTS、AI_WORKFLOW、RELEASE 和 loader README 都能指向同一套发布路径。
- 新对话中的 AI 只读本项目文档即可知道：
  - 原版 Visual Novel 脚本在哪里。
  - IGS 主程序源码在哪里。
  - 最终要发布成什么 JSON。
  - 发布前运行什么命令。
  - 私有仓库不能直接作为普通用户的远程 loader 源。
- 文档不得要求默认安装版实机验真；默认仍使用 fixtures 驱动的模拟测试。
- 文档不得把 IGS 默认改回奶龙工具箱 `project.json / latest / tavern helper` 发布壳。

## Assumptions

- IGS 继续保持独立 app 工程，不默认接回奶龙工具箱 `pack-project` 流程。
- 酒馆导入件采用“玉子手机式自动更新 loader”方案。
- 公开发布源在真正发布前再由用户确认：public 仓库、public release asset 或单独公开发布仓库。
