# v0.1.5 验收闸门策划书

版本：v0.1.5
日期：2026-06-12
状态：策划归档，待实施

## Summary

- 目标：先为 Visual Novel建立独立 app 验收闸门，让后续 AI 重构每一步都有可运行、可定位、可回归的检查。
- 范围：只建设 `app/` 自身的测试、模拟、校验、性能 smoke 和构建闸门；不接入奶龙工具箱 `pack-project / verify-project / validate / check-refs`。
- 推荐栈：`Vitest` + `jsdom` + `fake-indexeddb` + `Ajv` + `Vite library build`。
- 本机现状：已确认 `node v25.9.0`、`npm 11.12.1` 可用，未检测到 `pnpm/corepack`；实施时先保证 `npm run ...` 可跑，文档继续保留 `pnpm ...` 目标命令。

## 闸门设计

### Gate 0: environment

- 确认 Node 可用、依赖可安装、命令存在。
- 当前先以 `npm run gate` 为本机可执行入口。
- 后续安装 `pnpm` 后要求 `pnpm gate` 等价通过。

### Gate 1: structure

- 检查项目仍是独立 app 工程。
- 不得新增 `project.json`、`latest/`、`tavern helper/`。
- 不得把验收接回奶龙工具箱命令。

### Gate 2: static

- 检查 JS 语法。
- 检查 JSON 可解析。
- 检查 schema 可编译。
- 校验 `.vn-preset.json`、`.vn-pack.json`、provider、request-builder、prompt schema 基础结构。

### Gate 3: unit

- 覆盖纯逻辑模块：正文解析、时空栏正则、背景规则、角色情绪匹配、视觉模式切换、NAI request builder、prompt adapter、provider 启停。

### Gate 4: simulation

- 使用 fake TavernHelper、fake shujuku、fake DOM、fake IndexedDB 跑关键链路。
- 覆盖选项写入输入框再发送。
- 覆盖 shujuku 改表后刷新世界书。
- 覆盖本地资源刷新后保留。
- 覆盖生图段出现/消失切层。

### Gate 5: style-contract

- 检查稳定槽位、`.vn-` 前缀、`data-vn-*`、CSS 变量和 `settingsSchema` 桥接。
- 目标是保证换皮后 `设置 -> 阅读器` 仍可控制 UI。

### Gate 6: import-contract

- 检查合集按 `type` 分发到对应能力页。
- 拒绝全局 Mod 管理页。
- 拒绝单独总导入口。
- 拒绝快捷键预设导入。
- 拒绝 shujuku 模板导入。

### Gate 7: perf-smoke

- 使用合成大图包、大 shujuku 表、多 provider 禁用场景、移动端横屏视口做性能 smoke。
- 阈值先用 proposed default，后续实机测量后收紧。

### Gate 8: build-release

- 构建 `app/dist/vn.bundle.js`、`app/dist/vn.bundle.css`、`app/dist/manifest.json`。
- 检查产物存在、可加载、无真实密钥或隐私 fixture。

## 施工步骤

1. 在 `app/` 建立独立工程脚本：新增 `package.json`、`vitest.config.js`、测试 setup；脚本包含 `test`、`simulate`、`build`、`perf`、`gate`。
2. 建立测试基础设施：`tests/setup/` 放 fake DOM、fake TavernHelper、fake shujuku、fake IndexedDB；所有 fake 都只模拟契约，不连接真实酒馆或网络。
3. 建立 fixture 分层：`fixtures/tavern/`、`fixtures/shujuku/`、`fixtures/media/`、`fixtures/providers/`、`fixtures/imports/`、`fixtures/styles/`；fixture 禁止真实 API key、真实用户正文和隐私路径。
4. 建立 schema 校验：用 `Ajv` 校验 preset、pack、provider、request-builder、import-bundle；先覆盖当前已确认类型，后续功能新增时必须补 schema 或显式记录暂不校验。
5. 建立首批 P0 测试：`host` 输入发送、`scene` 时空栏和规则解析、`visual` 生图切层、`storage/media` 资源持久化、`data/shujuku` 写表并刷新世界书。
6. 建立 AI 友好失败信息：每个测试名使用 `gate:<模块>:<行为>`，失败信息指出应看哪个模块契约和 fixture。
7. 更新工作流文档：把每次实现后至少跑哪些 gate、失败如何定位、哪些命令不允许使用写入 `docs/AI_WORKFLOW.md`，项目记录只写当前项目 README。

## 命令契约

本机当前可执行目标：

```text
npm run test
npm run simulate
npm run build
npm run perf
npm run gate
```

文档目标命令保留：

```text
pnpm test
pnpm simulate
pnpm build
pnpm perf
pnpm gate
```

`gate` 顺序固定为：

```text
structure -> static -> test -> simulate -> perf -> build
```

前一层失败时停止，避免后续错误淹没根因。

## 通过标准

- 结构通过：项目内不存在奶龙工具箱发布壳，验收命令不调用 `pack-project`、`verify-project`、`validate`、`check-refs`。
- 静态通过：所有 JS 可被解析，所有 JSON/schema 可被解析，所有 fixture 不含密钥类字段样例。
- 单测通过：核心纯逻辑在无真实酒馆环境下可测，失败能定位到模块。
- 模拟通过：输入发送、shujuku 刷新、本地资源持久化、生图切层、导入分发至少各有 1 条 happy path 和 1 条失败/边界用例。
- 样式契约通过：稳定槽位存在，换皮 CSS 不破坏设置桥接；`data-vn-*` 和关键 CSS 变量可被测试读取。
- 性能 smoke 通过：大 fixture 不出现同步死循环；禁用 provider 不继续扫描或轮询；移动端横屏布局测试不产生关键层级缺失。
- 构建通过：`dist` 产物存在，manifest 指向正确文件，bundle 不包含测试 fixture、真实密钥或本机绝对路径。

## Assumptions

- 验收闸门先服务重构，不要求一次覆盖全部未来功能。
- P0 不引入 Playwright；等阅读器真实 UI 可运行后，再增加浏览器截图和移动端视觉回归。
- P0 性能阈值使用 proposed default，只作为卡顿预警；等有真实页面和资源包后再用实测值替换。
- 本策划书只规划验收工程，不实现业务功能、不迁移 Visual Novel 功能、不接入 GitHub 发布。
