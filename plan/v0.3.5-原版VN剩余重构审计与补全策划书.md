# v0.3.5 原版 VN 剩余重构审计与补全策划书

## Summary

- 结论：截至 VN `v0.3.4`，原版 Visual Novel 还没有完全重构为 VN；阅读器主体、魔法棒入口、正文管线、基础设置面板、分页、拖拽、跨楼层和基础图片保存/重绘已经有代码与测试证据，但图像 provider、外部插图适配、DOM 图片探测、隐藏楼层兼容和 SillyTavern context fallback 仍未完全等价。
- 目标模式：奶龙工具箱 A/B 模式边界内的酒馆助手脚本 / JS 源码能力；本项目按项目级 `AGENTS.md` 例外执行，保持独立 `app/` + `loader/` 工程，不接回 `project.json/latest/archive/tavern helper` 发布壳。
- 下一版本建议：`v0.3.5` 先补齐原版 VN parity 缺口，不把 shujuku 表格页、选项浮窗、UI Skin 管理页、背景/立绘资源包等 VN 新增目标混入“原版 VN 未重构”清单。
- 风险级别：本策划书为 `R1` 文档/计划；后续实施涉及 provider、host、generated-images、visual-novel-ui 运行时，应提升到 `R3`，若触及 loader/dist 发布再提升到 `R4`。

## 结论：是否完全重构

没有完全重构。

已完成的部分集中在“用户可打开、可读、可翻页、可跨 AI 楼层、可保存/触发外部重绘、可使用原版设置面板”的阅读器核心体验。未完成的部分集中在“原版图像 API 的真实网络能力、外部插图适配器选择与检测、复杂 DOM/iframe 图片探测、隐藏楼层兜底和 SillyTavern context 兼容”。

旧 `window.VisualNovelBridge` 未挂回不列为缺口。证据是 VN API 契约明确使用 `window.VN` / `window.VisualNovel`，并写明不继续扩散旧 `VisualNovelBridge` 调试入口；这是重构命名边界，不是原版能力遗漏。

## 未重构 / 未完全重构清单

### P0-1. 内置 NAI 图像 API 真实链路未重构

**原版证据**

- 原版有真实模型拉取：`fetchModels()` / `fetchImageModels()` 会请求 `/models`，解析 `id/name/model/data/models` 等返回结构，见 E15。
- 原版 `testImageApi()` 在 NAI 模式下会调用 `generateImage()` 做真实生成测试，见 E15。
- 原版 `generateImage()` 覆盖 `/v1/images/generations` URL 归一化、headers、JSON body、图片响应、zip/octet-stream、base64、异步 `status_url/poll_url` 轮询，见 E15。

**VN 现状证据**

- `nai-provider.js` 的 `generate()` 直接返回 `{ ok:false, reason:'not-implemented' }`，见 E16。
- `bootstrapVN()` 默认没有 `options.imageGenerator` 就返回 `provider-not-enabled`，见 E17。
- `reader-host.js` 的 `fetch-image-models` 固定写入两个模型名，`test-image` 固定显示 fake provider 文案，见 E18。

**影响**

- 用户在 VN 设置页选择 `VN 内置 NAI API` 后，模型拉取和测试生成并没有对真实接口进行验证。
- 阅读器重绘在 `imageApi.mode === 'nai'` 时依赖外部注入 `options.imageGenerator`，默认导入 loader 的用户无法获得原版内置 NAI 生图能力。

**下一版施工落点**

- 在 `app/src/generated-images/providers/nai-provider.js` 实现真实 `generate()`、`poll()` 与响应归一化。
- 新增 `proposed: app/src/generated-images/image-api-client.js`，承载 `normalizeApiBase()`、`modelsUrl()`、`imageGenerationUrl()`、`fetchModels()`、`parseImageResponse()`、`pollImageTask()` 等纯函数，避免把网络细节塞进 `reader-host.js`。
- `core/bootstrap.js` 给 `createVisualNovelReaderHost()` 注入 `fetchImageModels(settings)` 与 `testImageApi(settings)` 回调；`reader-host.js` 删除固定 fake 分支，只负责渲染 loading/success/error 文案。
- `visual-novel-compat.generateImage()` 继续走 VN API，但默认应能调用内置 NAI provider，而不是只依赖 `options.imageGenerator`。

**验收**

- `npm run test`：新增 fake fetch 单测，覆盖 `/models` 成功/失败、同步图片、base64、zip/octet-stream、异步 poll 成功/超时。
- `npm run simulate`：设置页 `fetch-image-models` 不再出现“模拟 provider”文案；NAI 模式 `test-image` 使用 fake fetch 走真实解析链路。
- `npm run gate`。

### P0-2. 外部插图适配器选择与检测没有接入运行时

**原版证据**

- 原版 `detectImageExtension(options)` 会读取 `source.externalAdapter`，对 `chami`、`chatu8`、`auto` 分别选择适配器并统计图片/按钮/元数据，见 E19。
- 原版 `testImageApi()` 在非 NAI 模式下返回 `detectImageExtension(source)`，见 E19。

**VN 现状证据**

- VN 设置页保留了 `bridge.imageApi.externalAdapter` 下拉，含 `auto/chatu8/chami`，见 E18、E21。
- VN `resolveActiveProviders()` 在没有显式 providers 时直接返回全部 builtin providers，没有读取 `imageApi.externalAdapter`，见 E20。
- VN `provider-runtime.js` 当前只按传入 provider 列表逐个检测和提图，没有接收或读取 `imageApi.externalAdapter`，见 E32。
- `st-chatu8-provider.js` 与 `chami-provider.js` 的 `detect()` 只看 DOM selector，不读取用户选择，见 E21。
- 设置页 `test-image` 对外部扩展固定显示 fake 检测文案，见 E18。

**影响**

- 用户选择只检测 `st-chatu8` 或只检测 `chami` 时，运行时仍可能同时扫描两个 provider。
- `generated-images/CONTRACT.md:39-40` 要求“provider 禁用后不继续请求或扫描”，当前仅有 selector 检测，缺少启用/禁用与 adapter 选择过滤，见 E31。

**下一版施工落点**

- 修改 `resolveActiveProviders()` 为 `resolveActiveProviders(explicitProviders, builtinProviders, imageApi)`，按 `externalAdapter` 过滤：`chatu8` 只保留 `builtin.st-chatu8`，`chami` 只保留 `builtin.chami`，`auto` 保留当前自动检测。
- 在 `st-chatu8-provider.js` / `chami-provider.js` 中补 `providerType` 或 `adapterKey` 字段，避免靠 label 字符串判断。
- 给 `reader-host.js` 的 `test-image` 接入真实检测回调，返回 counts 与 selected adapter，而不是 fake 文案。

**验收**

- `npm run test`：单测覆盖 `externalAdapter='chatu8'` 时不调用 chami provider，`externalAdapter='chami'` 时不调用 st-chatu8 provider。
- `npm run simulate`：外部扩展检测返回实际 counts；切换 adapter 后 reader image state 只包含目标 provider 图片。
- `npm run gate`。

### P1-3. DOM 图片探测与重绘按钮根域未完整覆盖原版

**原版证据**

- 原版 `getMessageScopedRoots()` 会把消息内 iframe 的 document 加入扫描根，见 E22。
- 原版图片候选覆盖 `img[src]`、`img[data-src]`、`VIDEO`、`A[href]`、`backgroundImage`、`.tsp-*`、`data-image-id`、`data-location-hash`、generic `img[src^="blob:"]` 和 `img[src^="data:image"]`，见 E22。
- 原版 `findRegenButton()` 同样基于这些 scoped roots 查找 st-chatu8/chami 重绘按钮，见 E22。

**VN 现状证据**

- VN `host/tavern-helper-adapter.js` 的 `getMessageScopedRoots()` 只加入 message element 与 `.mes_text`，见 E23。
- VN `reader-image-service.js` 的 fallback `findScopedRegenerateButton()` 同样只加入 message element 与 `.mes_text`，见 E23。
- VN st-chatu8/chami provider URL 提取只取 `currentSrc || src`，见 E24。
- VN 代码中 `blob:` / `data:image` 只在保存 URL 可下载校验里出现，没有 generic DOM 图片 provider 证据，见 E24。

**影响**

- 如果外部插件把图片或重绘按钮放进楼层内 iframe，VN 可能收不到图片，也找不到对应重绘按钮。
- 如果图片只有 `data-src`、CSS `background-image`、`video`、`a[href]` 或普通 blob/data image 而没有 st-chatu8/chami class，VN 不会按原版方式纳入当前楼层图片池。

**下一版施工落点**

- 在 `host/tavern-helper-adapter.js` 提供可复用 `getMessageScopedRoots(message, doc?)`，覆盖非 VN 内部 iframe document，返回去重 roots。
- 新增 `proposed: app/src/generated-images/dom-image-candidates.js`，复刻原版 `rawImageUrl()`、`imageCandidateGroupKey()` 和 selector 分组策略；provider 与 reader-image-service 共用，不复制三份 DOM 扫描代码。
- 增加 `generic-dom-image-provider` 或在 provider-runtime 前置 generic DOM 提取，覆盖 generic `blob:` / `data:image` 图片。
- `reader-image-service.resolveRegenerateButton()` 与 `hostAdapter.findRegenerateButton()` 使用同一 scoped roots helper。

**验收**

- `npm run test`：单测覆盖 `data-src`、CSS `backgroundImage`、`video.src`、`a[href^="blob:"]`、generic data image。
- `npm run simulate`：fixture 构造楼层内 iframe，iframe 里放图片和重绘按钮；打开阅读器后能显示图片，点击重绘能命中按钮并完成 poll。

### P1-4. 隐藏楼层 `hide_state` 兜底未完整重构

**原版证据**

- 原版 `readHiddenMessageIdsFromTavernHelper()` 调用 `helper.getChatMessages('0-' + lastId, { hide_state: 'hidden', include_swipes: false })`，并将返回的 `message_id/mesid` 建成隐藏 ID Set，见 E25。
- 原版 `isHiddenByChatState()` 会同时看 chat entry 的 `is_system/is_hidden` 和隐藏 ID Set，见 E25。

**VN 现状证据**

- VN `isHiddenMessage()` 只检查 `is_hidden`、`isHidden`、`hidden` 或 DOM `is_hidden="true"`，见 E26。
- VN `tavern-helper-adapter.js` 没有 `hide_state` 检索证据，见 E26。

**影响**

- 如果 TavernHelper 普通 `getChatMessages('0-last')` 不带隐藏状态字段，但 `hide_state:'hidden'` 能返回隐藏楼层，VN 会比原版更容易把隐藏楼层纳入“最近 AI 楼层”或上一轮/下一轮候选。

**下一版施工落点**

- 在 `host/tavern-helper-adapter.js` 增加 `readHiddenMessageIds(helper)` 与 300ms 级缓存，逻辑对齐原版 E25。
- `normalizeMessage()` 或 `isHiddenMessage()` 引入 hidden ID Set，保证普通消息对象缺少 `is_hidden` 字段时仍能排除隐藏楼层。
- `core/bootstrap.js` 的 fallback `isVisibleAiTurn()` 保留字段判断；hostAdapter 提供的 normalized messages 必须已经含 `isHidden: true/false`。

**验收**

- `npm run test`：fake TavernHelper 普通消息不含 hidden 字段，额外 `hide_state:'hidden'` 返回某 ID；断言 `listMessages()` 中该 ID 的 `isHidden === true`，`getCurrentMessage()` 与 `getAdjacentMessage()` 跳过它。
- `npm run simulate`：阅读器 `prev-turn/next-turn` 在隐藏楼层夹在两条 AI 消息之间时仍按原版跳转。
- `npm run gate`。

### P2-5. `SillyTavern.getContext()` 兜底兼容未接入默认 host adapter

**原版证据**

- 原版 bridge 和 reader 都实现了 `getSillyTavernContext()`，会尝试 `window`、`parent`、`top` 上的 `SillyTavern.getContext()`，见 E27。
- 原版读取 message entry、lastId fallback 和 hidden state 时都可能使用 `ctx.chat`，见 E27。

**VN 现状证据**

- VN host 契约写明职责包括 `TavernHelper`、`SillyTavern DOM` 和魔法棒菜单，见 E28。
- VN 默认 `createTavernHelperAdapter()` 只读取 `globalObject.TavernHelper`；没有源码证据显示默认 adapter 使用 `SillyTavern.getContext()`，见 E28。

**影响**

- 在 `TavernHelper` 不可用但 `SillyTavern.getContext()` 可用的宿主环境中，原版仍可能读取 chat，而 VN 默认 adapter 会返回 `null` 或 `missing-tavern-helper`。

**下一版施工落点**

- 在 `host/tavern-helper-adapter.js` 增加 `getSillyTavernContext(globalObject)` 与 `getChatFromContext()`，作为 `getLastMessageId()`、`getMessageById()`、`listMessages()` 的 fallback。
- 仍保持 A/B 酒馆助手脚本优先：`TavernHelper` 可用时优先使用 TavernHelper，只有缺失或返回空时才走 context fallback。

**验收**

- `npm run test`：fake `SillyTavern.getContext().chat` 存在而 `TavernHelper` 缺失，断言 `getCurrentMessage()` 能返回最后一条可读 AI 消息。
- `npm run simulate`：无 TavernHelper、有 SillyTavern context 的环境可打开阅读器；若两者都缺失，返回明确 `missing-host-message-api` / `no-message`。
- `npm run gate`。

## 残余风险

- 真实 provider 网络验收不能依赖真实 API key；实施轮应使用 fake fetch 和 fixture，真实接口只在用户明确提供测试条件时执行。
