# v0.3.6 image 标签图位绑定与图片轮询修复施工图

## Summary

- 本施工图目标是把 IGS 阅读器的图片顺序从“DOM/provider 扫到什么就按什么排”改为“先按楼层正文里的 `<image>` / `image###...###` 建立图位，再把扫描到的真实图片填回对应图位”。
- 当前 IGS 已有“图片保留标签”设置，也已经能从原文抽出 `imageSource`，但 `imageSource` 没有继续传给阅读器图片服务，导致图位顺序与作者写在原文里的 `<image>` 顺序脱节。
- 本轮计划不新增 UI 控件，复用现有设置页的 `图片保留标签`。实现后用户能看到的变化是：第三个 `<image>` 对应的段落打开时，阅读器进度显示为 `[3/6 图]`，重绘和保存也绑定第三个图位，不再把扫描到的单张图当作第一张。
- 目标模式：奶龙工具箱 B. JS-Slash-Runner 纯 JS 脚本边界内的独立 IGS app 工程；本项目继续使用 `app/` + `loader/` 结构，不接入 `project.json/latest/archive/tavern helper` 发布壳。
- 风险级别：本策划书为 `R1` 文档计划；实现轮会修改 `scene`、`api`、`generated-images`、`visual-novel-ui`、`host` 和模拟测试，应按 `R3` 执行；若同步 loader/dist、版本标签和远程发布，应提升到 `R4`。

### 未确认项

- 没有源码证据证明所有外部插图扩展都会把 `<image>` 的标题、hash 或顺序写到 DOM 元数据里；因此实现轮必须设计“精确匹配优先，当前图位兜底，旧顺序兜底最后使用”的分层策略。
- 若页面只暴露一张 DOM 图片且没有 `data-image-id`、`data-location-hash`、按钮元数据或当前正文图位，程序无法从图片 URL 本身反推出它属于第几个 `<image>`；这种情况只能通过当前正文图位或旧顺序规则处理，不能写成已确认能力。

## 当前问题结论

当前 IGS 的图片顺序问题不是“图片保留标签没有被提取”，证据 E07 和 E08 显示它已经被提取为 `imageSource`。

真正断点在 `visual-novel-compat.js` 到 `reader-image-service.js` 之间：`imageSource` 没有继续进入图片服务，`collectMessageImages()` 只能拿 DOM/provider/cache/scene 图片数组，再按扫描顺序合并。阅读器随后把 `imageState.currentIndex` 当作当前图序显示，所以当 DOM 里只扫到第三张图时，它会被放在数组第 1 位，显示成 `[1/1 图]` 或第一张背景。

原版 VN 的做法有证据可循：先从文本里提取图片块，再在 `parseRawSource()` 中给正文段落写入 `imgIdx`，渲染和重绘都使用这个 `imgIdx`。IGS 下一版应迁移这套“正文段落绑定图位”的语义，同时保留 v0.3.5 已完成的 provider、NAI、iframe、隐藏楼层和 context fallback 能力。

## 施工步骤

### 1. 新增图位解析层

在 `app/src/scene/image-slots.js` 新增纯函数，避免把图位解析塞进图片服务。

需要导出：

- `parseImageSlots(raw, imageSource, sourceFilter)`
- `extractImageBlocks(source)`
- `buildSegmentImageMap(raw, segmentTexts, imageSlots)`
- `normalizeImageSlotTitle(rawBlock)`
- `buildImageSlotHash(rawBlock)`

`parseImageSlots()` 输出结构：

```js
{
    slotIndex: 0,
    title: '望月的抗拒背影',
    tagName: 'image',
    rawBlock: '<image>...</image>',
    promptText: 'image###Scene:SFW,...###',
    locationHash: 'sha1-or-stable-hash',
    source: 'image-tag'
}
```

约束：

- 支持 `<image>...</image>`。
- 支持没有外层 `<image>` 的 `image###...###`。
- 标题取图片块内第一行方括号文本；没有标题时使用 `图 1`、`图 2` 等内部 fallback。
- `locationHash` 对规范化后的 `rawBlock` 或 `promptText` 生成稳定短 hash；实现不依赖 Node crypto 时可用项目内轻量 hash，必须有单测固定。
- 只有 `sourceFilter.enabled !== false` 且 `imageIncludeTags` 解析后非空时启用图位绑定；图片保留标签清空或标签筛选关闭时，走旧 DOM/provider 顺序 fallback。

### 2. 在正文 payload 中传递图位

修改 `app/src/scene/message-source.js`：

- `buildVisualNovelTextPayload()` 在返回值中增加 `imageSlots` 和 `segmentImageSlots`。
- `imageSlots` 从 `raw` 和 `strictPayload.imageSource` 构建。
- `segmentImageSlots` 使用与原版 `parseRawSource()` 对齐的语义：按 `<image>` 在原始文本里的出现位置，把正文段落映射到 `slotIndex`。
- 不删除原有 `imageSource`、`formattedRaw`、`formattedText` 字段，保证旧测试和外部调用不破坏。

段落映射规则：

- `<image>` 前的文本段绑定当前图片计数。
- 遇到 `<image>` 后图片计数加一。
- 计数超出 slot 数时夹到最后一个 slot。
- 原文没有可定位 `<image>` 但有 `imageSource` 时，按段落数量均匀分配到图位，行为对齐原版 `parseRawSource()` 的无内嵌图位 fallback。
- 没有图位时返回空数组，让下游走旧顺序。

### 3. 把图位传入阅读器图片服务

修改 `app/src/api/visual-novel-compat.js`：

- `buildReaderPayload()` 保留 `visualNovelText.imageSlots` 和 `visualNovelText.segmentImageSlots`。
- `enrichReaderPayload()` 调用 `collectMessageImages()` 时传入：
  - `imageSource`
  - `imageSlots`
  - `segmentImageSlots`
  - `textIndex`
  - `preferredImageIndex`
- `preferredImageIndex` 由当前正文页推导，不再在进入图片服务前写死为 `0`。
- `startAtEnd === true` 时，先取最后一个正文段落 index，再通过 `segmentImageSlots` 算出对应图位，不再把 `Number.MAX_SAFE_INTEGER` 直接当图片 index。

### 4. 重构 imageState 为槽位优先

修改 `app/src/generated-images/reader-image-service.js`：

- `collect()` 接收 `imageSlots` 和 `preferredImageIndex`。
- 当 `imageSlots.length > 0` 时，先生成长度等于图位数的 `slotImages`。
- provider/cache/scene 图片只负责填充 slot，不改变 slot 顺序。
- `buildImageState()` 输出增加：
  - `slots`
  - `images`
  - `count`
  - `currentIndex`
  - `currentUrl`
  - `displayUrl`
  - `unboundImages`
- `images` 在有图位时应与 `slots` 同长，每项可有空 `url`；`currentIndex` 始终表示图位 index。
- 当前 slot 没有 URL 时，`displayUrl` 按原版 E23 先向后再向前找最近可用 URL；`currentIndex` 不因此改变。
- 没有图位时保持当前 v0.3.5 的 provider/cache/scene 顺序，作为 fallback。

填充优先级：

1. 缓存中已有相同 `slotIndex`、`locationHash` 或 `imageId` 的图片。
2. DOM/provider 候选带 `slotIndex`。
3. DOM/provider 候选带 `locationHash` 且匹配图位。
4. DOM/provider 候选带 `imageId` 且缓存或按钮元数据能对应图位。
5. 候选数量等于图位数量时，按候选 DOM 顺序填充对应图位。
6. 候选数量少于图位数量且存在 `preferredImageIndex` 时，把首个未绑定候选填到当前图位。
7. 仍无法归属的候选进入 `unboundImages`，不抢占第 1 图位。

### 5. 扩展 DOM 图片候选元数据

修改 `app/src/generated-images/dom-image-candidates.js`：

- `collectDomImageCandidates()` 返回 `imageId`、`locationHash`、`slotIndex`、`buttonIndex`、`order`、`adapterKey`。
- 对 `data-image-id`、`data-location-hash`、`data-slot-index`、`data-image-index` 做统一读取。
- `imageCandidateGroupKey()` 保留现有去重能力，但不要把元数据丢掉。
- `findDomRegenerateButtons()` 同样返回按钮候选对象，或新增 `collectDomRegenerateButtonCandidates()`，让重绘按钮可按 `locationHash/imageId/slotIndex/order` 匹配。

### 6. 让 provider 携带图位信息

修改 `provider-runtime.js` 和内置 provider：

- provider `extractImages()` 返回值允许包含 `slotIndex`、`locationHash`、`imageId`、`order`。
- `st-chatu8-provider.js`、`chami-provider.js`、`dom-generic-provider.js` 复用 `dom-image-candidates` 的扩展候选信息。
- `collectProviderImages()` 不丢弃这些字段。
- `uniqueImages()` 去重时保留第一份 URL，同时合并更完整的 slot 元数据。

### 7. 分清正文 index 和图片 index

修改 `reader-host.js`：

- 当前正文页继续叫 `currentIndex` 或新增 `textIndex`。
- 当前图位统一使用 `imageIndex`。

- `buildImageActionContext()` 传给保存、重绘、生成的 `currentIndex` 保留兼容，但新增明确 `textIndex` 和 `imageIndex`；服务端逻辑优先使用 `imageIndex`。

### 8. 修复重绘按钮与轮询保持槽位

## 数据流与接口契约

目标数据流：

```text
raw message
-> buildVisualNovelTextPayload()
-> imageSource + imageSlots + segmentImageSlots
-> enrichReaderPayload()
-> collectMessageImages({ imageSlots, preferredImageIndex })
-> imageState.slots/currentIndex/currentUrl/displayUrl
-> reader-host background/progress/save/regen
```

`imageSlots` 契约：

```js
[
    {
        slotIndex: 0,
        title: '望月的抗拒背影',
        tagName: 'image',
        rawBlock: '<image>...</image>',
        promptText: 'image###...###',
        locationHash: 'stable-hash',
        source: 'image-tag'
    }
]
```

`imageState` 契约：

```js
{
    ok: true,
    messageId: 123,
    slots: [
        {
            slotIndex: 0,
            title: '望月的抗拒背影',
            locationHash: 'stable-hash',
            url: 'https://example.com/slot-1.png',
            providerId: 'builtin.chami',
            source: 'provider-dom'
        }
    ],
    images: [],
    unboundImages: [],
    count: 6,
    currentIndex: 2,
    currentUrl: 'https://example.com/slot-3.png',
    displayUrl: 'https://example.com/slot-3.png',
    signature: '...'
}
```

兼容约束：

- 没有图位时，`imageState.images` 保持旧数组语义。
- 有图位时，`images` 可作为 `slots` 的兼容镜像，但 `currentIndex` 表示图位 index。
- `currentUrl` 表示当前图位自己的 URL；`displayUrl` 表示实际可显示 URL。当前图位没有 URL 时，`currentUrl` 为空，`displayUrl` 可来自最近可用图。
- 旧调用只传 `currentIndex` 时继续可用，但内部必须优先使用 `imageIndex`。

- 用户样例的 6 个 `<image>` 标题能按 E25 顺序解析。
- 第 3 个 `<image>` 不会因为 DOM 只扫到一张图而变成第 1 图。
- `imageState.count` 在有图位时等于 `<image>` 图位数量。

- `reader-host` 进度显示 `[3/6 图]`。
- 当前 slot 没有 URL 时不丢 slot，总数仍稳定；实际显示 URL 允许按原版规则找附近可用图。
- 多余 DOM 图片不会插到 slot 0 抢位置。
- 重绘、保存、NAI 生成和外部 provider 轮询都使用 `imageIndex`。
- 现有 v0.3.5 回归不退化：NAI、外部插图适配、DOM generic 图片、iframe `data-src`、隐藏楼层、SillyTavern context fallback 仍通过测试。

## 残余风险

- 外部插件如果不暴露 `data-image-id`、`data-location-hash` 或 slot index，且同一楼层同时出现多张无法区分的 DOM 图片，程序只能按候选顺序或当前图位 fallback，无法凭 URL 自己推断作者原始图位。
- `locationHash` 需要在不同换行、空格和 HTML 注释差异下保持稳定；实现轮必须用 fixture 固定 hash 输入输出。
- 如果用户把“图片保留标签”改成不包含实际图片标签，图位绑定应关闭并回到旧扫描顺序，这属于预期行为，不应当悄悄强制解析 `<image>`。
